# Frontend Skill

## Overview

The public-facing storefront is a Next.js 14 app using the **Pages Router** (not App Router) with JavaScript.

## Architecture

```
frontend/
  pages/              # File-based routing
    index.js          # Redirects to /landing
    landing.js        # Landing page (reads landing.json)
    [slug].js         # Dynamic page for any slug (reads {slug}.json)
    _app.js
  components/
    sections/         # JSON-driven section components (Shopify-like pattern)
      SectionRenderer.js
      registry.js     # Maps type strings тЖТ React components (12 types)
      icons.js
      Navbar.js, Hero.js, Collections.js, About.js, Featured.js, Gallery.js, Contact.js
      ContentImage.js, Footer.js, MaterialShowcase.js, ColorPicker.js, ServiceProcess.js
    clone/            # Web clone section components
  config/
    pages/            # Page JSON configs тАФ auto-generated by backend
      landing.json, about.json, products.json, service.json, ...
  hooks/              # Custom React hooks (useReveal.js)
  lib/                # API helpers and utilities
  utils/              # Pure utility functions
  styles/             # Global CSS (globals.css with Tailwind directives)
  public/             # Static assets
  next.config.js      # Turbopack aliases, image config, API rewrites
```

## Styling Stack

- **Tailwind CSS v3** via `@tailwind base/components/utilities` in `globals.css`
- **PostCSS** + **Autoprefixer** configured in `postcss.config.js`
- `tailwind.config.js` scans `pages/` and `components/` for class usage

## Web Clone Pattern

When cloning external pages (via the `perfect-web-clone-skill`), follow this structure:

### Page file
- Create `pages/<name>.js` that imports numbered section components
- Each section maps 1:1 to a chunk from the extraction pipeline

```jsx
// pages/clone.js
import Section1 from '../components/clone/Section1';
import Section2 from '../components/clone/Section2';
// ...
export default function ClonePage() {
  return (
    <main>
      <Section1 />
      <Section2 />
    </main>
  );
}
```

### Section components
- Place in `components/<clone-name>/Section<N>.js`
- Each section is self-contained: all data, styles, and icons are inline
- Include a JSDoc block at the top describing what the section replicates, its captured dimensions, background color, and layout

### Hybrid styling approach
- **Inline `style={{}}`** for pixel-exact values from the source site: colors (`rgb(247, 226, 206)`), font sizes (`11px`), letter spacing, exact heights/widths
- **Tailwind classes** for layout: `flex`, `items-center`, `w-full`, `gap-x-5`, responsive breakpoints (`lg:flex`, `lg:hidden`)

### Inline SVG icons
- Define SVG icons as small React function components within the section file
- Do NOT add icon library dependencies (lucide-react, heroicons, etc.)
- Use `aria-hidden="true"` on decorative icons

### Responsive design
- Desktop: `lg:flex` (1024px+)
- Mobile: `lg:hidden` with separate layout
- Mobile navigation: hamburger menu with `useState` toggle and slide-down drawer

### Interactive effects
- Parallax: lightweight scroll-based implementation using `useRef` + `useEffect` + passive scroll/resize listeners
- Avoid heavy parallax libraries; calculate offset from viewport position directly

## JSON-Driven Section/Block System (Landing Page)

The landing page uses a Shopify-like config-driven architecture:

```
config/pages/landing.json тЖТ SectionRenderer тЖТ registry тЖТ Section Components
```

### Config structure (`config/pages/<page>.json`)
- **`page`**: metadata (title, description, bodyClass)
- **`order`**: flat array of section keys тАФ controls render order
- **`sections`**: map of section configs, each with `type`, `settings` (scalars), `blocks` (repeatable children)

### Key files
- `components/sections/SectionRenderer.js` тАФ iterates `order`, resolves components from registry
- `components/sections/registry.js` тАФ maps type strings to React components
- `components/sections/icons.js` тАФ SVG icon registry (JSON references icons by string name)
- `components/sections/*.js` тАФ section components accepting `{ id, settings, blocks }` props

### Current registered section types (registry.js)
`navbar`, `hero`, `collections`, `about`, `featured`, `gallery`, `contact`, `content-image`, `footer`, `material-showcase`, `color-picker`, `service-process`, `why-choose-us`

### `transformPageConfig.js` тАФ SSR child block transformation
`frontend/lib/transformPageConfig.js` is the critical middleware between the raw API response and section components. It transforms `block.data.*` arrays into the `blocks: [{ type, settings }]` child arrays that section components consume.

The mapping is driven by `ARRAY_BLOCK_MAP` тАФ a lookup table that mirrors `backend/src/config/blockJsonMapping.ts`:

```javascript
const ARRAY_BLOCK_MAP = {
  'content-image':    [{ dataKey: 'buttons',      blockType: 'content-button' }],
  collections:        [{ dataKey: 'products',     blockType: 'product-card' }],
  about:              [{ dataKey: 'stats',         blockType: 'stat' }],
  footer:             [
    { dataKey: 'infoLines',   blockType: 'footer-line' },
    { dataKey: 'socialLinks', blockType: 'footer-social' },
  ],
  // ... all block types with child arrays
};
```

Note: `navbar` was removed from ARRAY_BLOCK_MAP тАФ it now uses `menuHandle` to reference a Menu collection instead of inline `links` array.

**When adding a new block type with child items** (e.g., `steps`, `features`, `items`): add an entry to `ARRAY_BLOCK_MAP` in `transformPageConfig.js`, otherwise the SSR page will render the section with `blocks: []` and child items will be invisible.

**Data flow for SSR pages** (`landing.js`):
```
MongoDB тЖТ API /theme/pages/landing тЖТ transformPageData() тЖТ getNestedBlocks() тЖТ section components
```
The JSON config files (`frontend/config/pages/*.json`) are used only by static pages (`[slug].js`). The landing page always fetches live from the API.

### Adding a new section type
1. Create the component in `components/sections/NewSection.js` accepting `{ id, settings, blocks }`
2. Register it in `registry.js`
3. Add its field schema to `backend/src/config/blockFieldDefs.ts`
4. Add its mapping to `backend/src/config/blockJsonMapping.ts`
5. **Add an entry to `ARRAY_BLOCK_MAP` in `frontend/lib/transformPageConfig.js`** (if the block has child arrays)

**Do NOT** manually edit `config/pages/*.json` тАФ these are auto-generated by the backend. Edit via the theme editor admin or seed scripts.

### Multi-page routing
- `pages/[slug].js` handles all dynamic pages using **SSG** (`getStaticProps` + `getStaticPaths`)
- `getStaticPaths` reads all `config/pages/*.json` files, excluding `landing` (which has its own dedicated `pages/landing.js`)
- `pages/materials/[id].js` also uses SSG with `fallback: false` тАФ no ISR/blocking fallback
- Do NOT use `getServerSideProps` for these pages; they are fully static

### Font CSS variables
- Page-level fonts (`displayFont`, `bodyFont`) are injected as CSS variables via a `<style>` tag inside `<Head>`, not as inline `style={{}}` on the container div
- Default font for both display and body is `Cormorant` (not Outfit for body)

```jsx
// CORRECT тАФ inject into <Head>
const fontVarsStyle = `:root { --font-display: "${displayFont}", serif; --font-body: "${bodyFont}", sans-serif; }`;
<Head>
  <style>{fontVarsStyle}</style>
</Head>

// WRONG тАФ inline style on div does not reliably propagate to portals/iframes
<div style={{ '--font-display': '...', '--font-body': '...' }}>
```

### Live preview via postMessage
- Admin sends preview config to the frontend iframe via `window.postMessage({ type: 'PREVIEW_CONFIG', config })`
- Frontend `[slug].js` listens for this message and renders the config without saving
- This mirrors the `buildPreviewConfig.js` logic on the client side

## Custom Hooks

- **`useReveal`** (`hooks/useReveal.js`): IntersectionObserver-based scroll reveal. Returns a ref; attach to section root. Adds `.revealed` class when element enters viewport. Used with `.reveal`, `.reveal-left`, `.reveal-right`, `.stagger-children` CSS classes defined in `globals.css`.

## Key Conventions

1. **No TypeScript** тАФ all frontend code is plain JavaScript
2. **Pages Router** тАФ file-based routing under `pages/`
3. **Use path aliases** тАФ `@/components`, `@/lib`, etc. (configured in `next.config.js`)
4. **API calls** go through the `/api/*` proxy rewrite to the Express backend
5. **Image optimization** тАФ use Next.js `<Image>` for project images; for cloned pages, use standard `<img>` with source URLs
